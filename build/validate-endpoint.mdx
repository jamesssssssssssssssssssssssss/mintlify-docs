---
title: "The validate endpoint: peek and consume"
description: "How the two-call peek-then-consume design works, why it exists, and how to wire it correctly into a wrapper-plus-webhook flow."
---

Most flows need to validate a token in two places. A pre-render wrapper checks it before the form ever loads. A post-submit webhook checks it again before the response is accepted. One call that burns the token on read would fail the second check for a perfectly legitimate respondent. That is why the endpoint is split: peek is idempotent, consume burns.

<Warning>
Confirm this two-call design with your engineering team before you wire any connectors. Every wrapper-plus-webhook integration in the VerifyYou pattern depends on peek being side-effect-free and consume being the single burn point. Building both halves against the same endpoint would silently reject legitimate respondents.
</Warning>

## The two calls

### GET /v3/survey/peek

Read the current state of a token. Nothing changes. Call it as many times as you need.

```http
GET /v3/survey/peek?vyt=TOKEN
Authorization: Bearer pk_val_FLOW_RAND
```

Response:

```json
{
  "verified": true,
  "reason": "ok"
}
```

`verified` is `true` only when the token represents a completed, passing liveness and uniqueness check. `reason` is always present: `"ok"` when the token is valid and unused, a short machine-readable string when it is not — for example `"expired"`, `"not_verified"`, `"wrong_flow"`, or `"already_consumed"`.

This call is safe from a pre-render wrapper, a middleware function, or any entry gate before your form loads. Nothing changes in our database. The same peek call from a webhook five minutes later returns the same result — unless consume fired in between, in which case `reason` becomes `"already_consumed"`.

### POST /v3/survey/consume

Burns the token. Call this exactly once, at the moment you decide to accept a response.

```http
POST /v3/survey/consume?vyt=TOKEN
Authorization: Bearer pk_val_FLOW_RAND
```

Response:

```json
{
  "consumed": true
}
```

If the token is already consumed, expired, or does not belong to the Flow the key is scoped to, `consumed` will be `false`. Treat anything other than `{ "consumed": true }` as a rejection.

## Authentication: the per-Flow publishable key

Both calls use a `pk_val_` key, not your secret `sk_` key. This is a per-Flow publishable validation key. It can only check tokens for one specific Flow. It cannot read PII, create sessions, or take any other action. That makes it safe to hold in a survey platform's server-side credential store or in an environment variable on a lightweight wrapper function.

<Warning>
"No key at all" is not a safe fallback. An unauthenticated consume endpoint enables a token-burn denial-of-service attack — anyone who can observe a token URL can burn it before the real respondent submits — and creates an enumeration oracle. Always require the `pk_val_` key.
</Warning>

Generate a `pk_val_` key in the VerifyYou dashboard, scoped to the Flow you are building against. Rotate it the same way you would any API credential: invalidate the old one in the dashboard, update your environment variable, redeploy.

## Token security properties

- **Flow-bound.** A token minted for Flow A returns `{ "verified": false, "reason": "wrong_flow" }` when peeked against Flow B's key. You cannot accidentally accept a token from a different study.
- **Time-limited.** Tokens expire after approximately 30 minutes. A respondent who completes the liveness check and then sits idle before submitting will arrive with an expired token. Your failure path should tell them to re-verify.
- **Hashed at rest.** We store only the hash, not the raw token string. A database read from our side cannot produce a working token.
- **Rate-limited.** Repeated peek or consume calls for non-existent tokens are rate-limited to prevent brute-force enumeration.
- **Constant-time comparison.** Token lookups use constant-time comparison to prevent timing attacks.

## The atomic consume pattern

If you control your own database, the safest pattern is to replicate the consume semantics locally rather than relying purely on the API round-trip. A representative SQL sketch:

```sql
UPDATE tokens
SET used_at = now()
WHERE token_hash = $1
  AND flow_id   = $2
  AND used_at   IS NULL
  AND expires_at > now()
RETURNING verified, external_id;
```

If this UPDATE returns a row, the token was valid and is now burned. If it returns nothing, the token did not exist, was already consumed, or has expired. This is exactly the logic our `/v3/survey/consume` endpoint runs on our side. Running it in your own store as well gives you an offline audit trail and eliminates the double-accept race condition that would occur if two webhook deliveries fired simultaneously for the same response.

## Putting it together: a wrapper-plus-webhook flow

<Steps>
  <Step title="Respondent arrives at your wrapper page">
    Your server calls `GET /v3/survey/peek?vyt=TOKEN`. If `verified` is `false` or `reason` is anything other than `"ok"`, render an error page and stop. Do not load the form. The token is not burned.
  </Step>
  <Step title="Respondent fills out the form and submits">
    Your survey platform receives the submission. The `vyt` token traveled through the form as a hidden field or a URL parameter.
  </Step>
  <Step title="Webhook fires">
    Your webhook handler calls `GET /v3/survey/peek?vyt=TOKEN` again to revalidate. Because peek does not burn, this returns the same result as step one — provided consume has not fired yet.
  </Step>
  <Step title="Accept or reject">
    If peek returns `{ "verified": true, "reason": "ok" }`, call `POST /v3/survey/consume?vyt=TOKEN`. If consume returns `{ "consumed": true }`, accept and record the response. Otherwise delete or flag it.
  </Step>
</Steps>

The two peek calls serve different purposes. The first stops a respondent from loading the form with a bad token at all. The second is the line of defense at the moment of truth. Only one consume ever fires per legitimate respondent, at the point of acceptance.

## Error reference

| `reason` value | What it means | What to do |
|---|---|---|
| `ok` | Token is valid and unused | Proceed |
| `expired` | Token TTL elapsed | Ask the respondent to re-verify |
| `already_consumed` | Token was already burned | Reject; do not accept a duplicate response |
| `wrong_flow` | Token belongs to a different Flow | Reject; likely a misconfigured link |
| `not_verified` | Liveness or uniqueness check did not pass | Reject and screen out |

<Note>
The `reason` field on peek tells you the full story: `"ok"` means valid and unused, `"already_consumed"` means a prior consume call succeeded. Always check `reason`, not just `verified`, before deciding to call consume.
</Note>
