The full build is smaller than it looks. Once you see the shape, every survey tool and recruitment source is just a slot in the same two-leg frame. This page describes that shape from the inside, for engineers building and maintaining it.

## What we have today vs. what is new

**Today (v3 API, live)**

- `POST /v3/initialize` starts a hosted liveness and uniqueness session and returns a URL. On completion the respondent is forwarded to `ORIGIN + RETURN_PATH?vyt=TOKEN&vyc=1` â€” or `vyc=0` on failure. `pass_params` carries participant identifiers (for example `PROLIFIC_PID`) through on the return URL.
- `GET /v3/confirmations/TOKEN` (secret `sk_` key) confirms a token. `POST /v3/confirmations/TOKEN/lock` burns it.
- A no-code share link (`app.verifyyou.com/verification/ID`) and a success redirect URL in the dashboard. A failure or screen-out redirect does not exist yet.

**New (to build â€” described in present tense below)**

- `GET /v3/survey/peek?vyt=TOKEN` and `POST /v3/survey/consume?vyt=TOKEN`, the split validate/burn pair. See [/build/validate-endpoint](/build/validate-endpoint).
- A per-Verification publishable validation key (`pk_val_`).
- A failure/screen-out redirect on the verification itself.
- Per-verification config stored server-side.
- Activity-log enrichment and CSV export.

## The shared core

Five pieces, built once, reused by every connector family.

<CardGroup cols={2}>
  <Card title="pk_val validation key" icon="key">
    A per-Verification publishable key that can peek and consume tokens but cannot read PII or create sessions. Safe to store in a survey tool's server-side credential store. Revocable per Verification without rotating the main secret key.
  </Card>
  <Card title="peek / consume split" icon="code-branch">
    `peek` is idempotent: call it at entry, call it again from a webhook â€” the token stays alive. `consume` burns it exactly once, only when a real completion is accepted. The split lets you enforce at the gate and backstop with a webhook without double-burning.
  </Card>
  <Card title="Failure redirect" icon="arrow-turn-down">
    A screen-out URL stored per verification. When `vyc=0` arrives on the return URL, the respondent is forwarded there before the survey ever loads. Today this must be handled in wrapper logic; the dashboard field is the new build.
  </Card>
  <Card title="Per-verification config" icon="sliders">
    A server-side record per verification storing: survey platform, recruitment platform, survey URL or ID, screen-out URL, and the researcher's connected platform API token for webhook auto-delete. Connectors read this config; researchers set it once.
  </Card>
  <Card title="Activity log + CSV export" icon="table-list">
    Enriched per-session event log â€” initialized, verified, consumed, webhook fired, response deleted â€” with timestamps and the external tracker value. Exportable as CSV for audit and reconciliation.
  </Card>
</CardGroup>

## Connector families

There are four families. A platform slots into one or more of them. They do not depend on each other, which is what makes the build scale: adding a new survey tool means writing one adapter, not touching gate logic.

### Family A â€” Single-use link or credential

**Platforms: Qualtrics Distributions, Alchemer OTP pool, MTurk qualification**

The token is embedded into a single-use URL or credential that the survey platform issues and controls. The platform enforces exactly-once access natively; VerifyYou's job is to generate that credential and deliver it to the platform's API after verification passes.

One shared credential-delivery step at the end of the VerifyYou flow. Per-platform adapters call the platform's distribution or OTP API. No entry-gate wrapper needed â€” the platform's own link mechanism is the gate.

<Note>
Qualtrics carries a known timing edge: the Web Service element fires before the response is fully committed. Qualtrics Workflows Web Service tasks natively support DELETE, so the auto-delete backstop can live entirely inside Qualtrics â€” a Workflow with a condition (`verified != true`) triggers a Web Service DELETE call carrying `X-API-TOKEN`, no shared receiver required. The shared webhook receiver (Family C) remains the generic path for platforms that cannot do this natively; for Qualtrics it is optional. The single-use Distribution link is still the primary gate.
</Note>

<Note>
Alchemer OTP pool: confirm live whether an OTP burns on login-click or only on completion before marking this gate as closed.
</Note>

<Note>
MTurk qualification: after verification passes, grant the worker a qualification via `AssociateQualificationWithWorker`. The main HIT must set `ActionsGuarded=DiscoverPreviewAndAccept` on its qualification requirement so it is hidden from workers who have not yet been granted â€” unqualified workers cannot even discover or preview it.
</Note>

### Family B â€” Wrapper or embed entry gate

**Platforms: Typeform, Jotform, Tally, SurveyMonkey, Google Forms, any URL-addressable form**

A hosted wrapper page intercepts the respondent before the survey loads. It calls `peek` with the `vyt` from the URL and only forwards to the survey if the token is valid. If `vyt` is missing or fails, the respondent hits the screen-out path.

One hosted wrapper service â€” a small edge function or Next.js page â€” plus thin per-platform adapters that know how to construct the survey entry URL with the token passed through. The wrapper logic is shared; only the URL construction differs per platform.

<Note>
The wrapper is the right home for the failure redirect. When `vyc=0` arrives from VerifyYou, the wrapper reads the screen-out URL from per-verification config and redirects there before the survey loads.
</Note>

### Family C â€” Webhook auto-delete backstop

**Platforms: Typeform, Jotform, Tally, SurveyMonkey, Google Forms (via Apps Script), Qualtrics (optional, native alternative available)**

When a respondent submits the survey, the platform fires a completion webhook to a shared VerifyYou receiver. The receiver reads the `vyt` hidden-field value, calls `peek`, and if the token is missing, invalid, or already consumed, calls the platform's delete-response API to remove the submission before it reaches any CSV or dashboard.

One shared webhook receiver endpoint. Per-platform adapters implement signature verification (each platform signs its webhooks differently) and the delete-response API call. Google Forms has no native webhook; an Apps Script snippet in the form's bound script handles the submission event and posts to the shared receiver.

`consume` is called by the receiver only after a valid response is accepted, so the token cannot be reused even if the webhook fires twice.

Family C is always paired with Family B for platforms that lack a single-use link mechanism. Qualtrics does not need the shared receiver at all â€” its own Workflows Web Service task can issue the DELETE directly, using the same condition-then-DELETE pattern described in Family A. The shared receiver stays available as an alternate path if a Qualtrics account prefers a centralized backstop.

### Family D â€” Direct call

**Platforms: Formspark, own application**

The destination code calls `peek` (or `consume`) directly before saving a response. No wrapper, no webhook adapter. Enforcement timing is effectively identical to a true gate.

Zero per-platform build. Document the two endpoints and the `pk_val` key. Formspark users add a custom action; app developers call the endpoint from their own handler.

## Recruitment hardening

Two recruitment sources need specific handling at the entry point, before the VerifyYou check starts.

**Prolific â€” Secure External URL JWT verification**

Prolific signs the redirect with a short-lived RS256 JWT so you can confirm the respondent actually came from Prolific, not a guessed link. The token arrives as a `prolific_token` URL parameter, expires in roughly two minutes, and verifies against the JWKS published at `api.prolific.com/.well-known/study/jwks.json`. The shared core verifies this JWT at entry before initializing a verification session. Details at [/recruitment/prolific](/recruitment/prolific).

**Cint â€” S2S postback**

Cint expects a server-to-server disposition call when a respondent completes, is screened out, hits quota, or fails quality â€” a `POST` to `https://s2s.cint.com/fulfillment/respondents/transition` with the RID and status code in the JSON body (`complete=5`, `screenout=2`, `quota=3`, `quality=4`). The shared core fires this call from the VerifyYou server, not from the respondent's browser, so it cannot be spoofed by someone intercepting the return URL. S2S runs alongside the browser redirect as a hybrid, not a replacement for it â€” it is authoritative when the two disagree, since browser redirects fail silently in panel webviews and S2S does not.

## Build order

<Steps>
  <Step title="Core infrastructure">
    Ship the `pk_val` key system, the `peek`/`consume` endpoint pair, the per-verification config store, and the failure redirect field. Everything else depends on these. See [/build/validate-endpoint](/build/validate-endpoint).
  </Step>
  <Step title="Family D (direct call)">
    No new code, just documentation. Validates the endpoint design against a real integration (Formspark) before building adapters on top.
  </Step>
  <Step title="Family C (webhook receiver)">
    Build the shared receiver and the first two per-platform delete adapters â€” Typeform and Jotform, the highest-volume platforms. The Google Forms Apps Script snippet ships here.
  </Step>
  <Step title="Family B (wrapper/embed gate)">
    Build the shared wrapper service. Wire per-platform URL construction adapters. Connect to the failure redirect from step 1.
  </Step>
  <Step title="Family A (credential delivery)">
    Build per-platform distribution adapters â€” Qualtrics first (Distributions API; auto-delete backstop via a native Workflow Web Service DELETE task, no shared receiver required), then Alchemer OTP pool (confirm burn timing before marking closed), then MTurk qualification.
  </Step>
  <Step title="Recruitment hardening">
    Add Prolific JWT verification and Cint S2S postback to the entry flow.
  </Step>
  <Step title="Activity log + CSV export">
    Enrich the event log and expose the export. Can ship in parallel with any step above once the per-verification config store from step 1 is live.
  </Step>
</Steps>
